1. Contexto Atual do Projeto
1.1 Estado funcional

O KeepCalm Web Browser já possui:

Navegador funcional com sidebar integrada (Security Lab)
Motor de auditoria com nuclei
Ferramenta manual Repeater (estilo Burp)
UI consistente (Graphite & Glass)
Execução de ferramentas via backend (Rust + Tauri)
1.2 Diagnóstico técnico
Pontos fortes
Arquitetura já separa UI e execução
Uso de ferramentas reais (não mockadas)
Experiência de uso fluida dentro do navegador
Base ideal para automação guiada
Gargalos críticos

O sistema atualmente:

❌ Não captura tráfego real (sem contexto)
❌ Ferramentas operam isoladas
❌ Não há pipeline de exploração
❌ Repeater depende de input manual artificial
1.3 Conclusão do contexto

O sistema possui capacidade de execução, mas não possui entrada de dados real nem fluxo operacional

2. Objetivo desta Fase
Objetivo principal

Implementar a camada de captura e reutilização de requisições HTTP (Traffic Inspector)

Resultado esperado

Ao final desta fase, o sistema deve:

Capturar automaticamente requisições do navegador
Armazenar de forma estruturada
Exibir no UI
Permitir envio direto ao Repeater
Servir como base para todas as ferramentas futuras
3. Princípios de Arquitetura
3.1 Separação de responsabilidades
UI (Tauri)
↓
Core (Rust)
↓
Capture Engine
↓
Storage
↓
Tools (Nuclei / Repeater / futuras)
3.2 Regra fundamental

Nenhuma ferramenta deve operar sem passar pelo modelo de dados comum

3.3 Padrão de dados único

Toda requisição deve seguir um modelo padronizado.

4. Modelo de Dados (obrigatório)
4.1 Estrutura de Request
#[derive(Clone, Serialize, Deserialize)]
pub struct HttpRequest {
    pub id: String,
    pub method: String,
    pub url: String,
    pub headers: Vec<(String, String)>,
    pub body: Option<String>,
    pub timestamp: u64,
}
4.2 Estrutura de Response
#[derive(Clone, Serialize, Deserialize)]
pub struct HttpResponse {
    pub status: u16,
    pub headers: Vec<(String, String)>,
    pub body: Option<String>,
    pub duration_ms: u128,
}
4.3 Estrutura completa
#[derive(Clone, Serialize, Deserialize)]
pub struct HttpTransaction {
    pub id: String,
    pub request: HttpRequest,
    pub response: Option<HttpResponse>,
    pub tags: Vec<String>,
}

11. Regras de Integração de Ferramentas
11.1 Toda ferramenta deve:
Receber dados do modelo padrão
Retornar dados estruturados
Não acessar UI diretamente
11.2 Proibição
Ferramentas manipulando estado global
Saída não estruturada sendo enviada ao UI
12. Critérios de Conclusão da Fase

A fase só é considerada concluída se:

 Requests são capturados automaticamente
 Requests aparecem no painel Traffic
 É possível enviar qualquer request ao Repeater
 Repeater funciona com dados reais
 Nuclei pode usar endpoints capturados
